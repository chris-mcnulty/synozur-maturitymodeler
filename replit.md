# Orion - Synozur Multi-Model Maturity Platform

## Overview
Orion is a fullstack JavaScript application providing a multi-model maturity assessment platform. It features dynamic routing for assessments, CSV-based model management, gated PDF result generation, benchmarking, and extensive administrative controls. The platform aims to offer insightful maturity assessments, aligning with Synozur's vision as "the Transformation Company." Orion also functions as a comprehensive OAuth 2.1/OpenID Connect identity provider for the Synozur ecosystem.

## User Preferences
- Uses SendGrid for email delivery (API key method, not Replit connector)
- Prefers seeing metrics on home pages when data is available
- Assessment dimensions are valuable and should be emphasized

## System Architecture
The application is built with a modern fullstack architecture, prioritizing a dark-mode UI.

### UI/UX Decisions
-   **Design**: Dark-mode-first aesthetic with a primary purple and accent pink color scheme, using the Inter font.
-   **Responsiveness**: Utilizes responsive gradient styling.
-   **Navigation**: Features a collapsible admin sidebar with hover tooltips for enhanced usability.
-   **Content Editing**: Unified inline Q&A editing via `UnifiedQuestionEditor` component for seamless modification of question text, type, answers, scores, improvement statements, and resource links directly within the ModelBuilder.
-   **Model Creation**: Guided 5-step `ModelCreationWizard` for new models, covering Basics, Dimensions, Questions & Answers, Maturity Scale, and Publish.

### Technical Implementations
-   **Frontend**: React, Vite, TypeScript, Wouter for routing, Shadcn UI for component styling.
-   **Backend**: Express.js, PostgreSQL, Drizzle ORM.
-   **Storage**: PostgreSQL for relational data, Google Cloud Storage for assets.
-   **Authentication**: Passport-based session management with tenant-scoped, four-tier role-based access control (`global_admin`, `tenant_admin`, `tenant_modeler`, `user`). Supports Microsoft Entra ID (Azure AD) SSO with PKCE flow, auto-provisioning, Azure AD tenant ID tracking, and admin consent URL generation for enterprise onboarding.
-   **UI/UX**: Dark-mode-first with primary purple and accent pink, Inter font. Responsive gradient styling. Collapsible admin sidebar with hover tooltips.
-   **Scoring System**: For 100-point scale models, scores are calculated as a **percentage of maximum possible** — `(totalScore / maxPossibleScore) × maxMaturityScale`. This correctly handles both 0-4 and 0-100 answer ranges. 500-point scale models average raw scores. Max possible per question is determined dynamically (max answer score for MC, 4 for normalized numeric/multi-select). The pure scoring engine lives in `server/services/scoring.ts` (`calculateAssessmentScore`); `POST /api/assessments/:id/calculate` in `server/routes.ts` pre-loads questions/answers and delegates to it. Unit tests in `tests/unit/scoring.test.ts`.
-   **Automated Tests**: Vitest + Playwright foundation under `tests/` (see `tests/README.md`). Unit tests cover the scoring engine and the AI provider registry; a Playwright smoke suite covers signup → assessment → results, plus an admin "edit question" leg gated on `E2E_ADMIN_USERNAME` / `E2E_ADMIN_PASSWORD`. Run with `npx vitest run` and `npx playwright test`. New features should add at minimum a unit test for any pure scoring/aggregation logic and a Playwright happy-path spec for new user-facing journeys.
-   **Monthly Digest Controls**: Two-level opt-out system. Tenant level: `monthlyDigestEnabled` boolean on `tenants` table (default `true`) — acts as a master switch that silences all users in the tenant regardless of individual settings. Configurable in Admin → Tenants via a dedicated toggle. User level: `monthlyDigestOptOut` boolean on `users` table (default `true` = opted out); users flip their own preference on the Profile page, admins can flip it in the User edit dialog. Bulk operations in Admin → Users panel: "Opt everyone out", per-tenant opt-out, "Opt everyone in", and "Safe opt-in (exclude tenants)" — the latter lets you opt-in all production users except a cherry-picked list of tenants (e.g. narrative, curriculum). Endpoints: `POST /api/admin/users/bulk-digest-setting` (optOut + optional tenantId), `POST /api/admin/users/safe-opt-in-except` (excludeTenantIds[]), `GET /api/admin/users/digest-stats`. Digest service (`server/services/digest-service.ts`) LEFT-JOINs tenants and skips users whose tenant has `monthlyDigestEnabled=false`. Migration: `migrations/0003_tenants_digest_enabled.sql`.
-   **Model Management**: CSV-driven import/export plus .model JSON format. ModelBuilder component with Overview, Structure, Resources, and Maturity Scale tabs. Model archiving preserves data while hiding from default views.
-   **Admin Content Experience**: Unified inline Q&A editing via `UnifiedQuestionEditor` component (client/src/components/admin/UnifiedQuestionEditor.tsx) — question text, type, answers, scores, improvement statements, and resource links are all editable inline in the ModelBuilder Structure tab with no modal dialogs. New models are created via `ModelCreationWizard` (client/src/components/admin/ModelCreationWizard.tsx) — a 5-step guided flow (Basics → Dimensions → Questions & Answers → Maturity Scale → Publish). The standalone Dimensions and Questions nav items have been removed from the Admin sidebar; all model content is managed through All Models → open model → Structure tab. The Content/Bulk Tools tab (ContentManagement.tsx) handles only CSV import/export and AI tool quick-links.
-   **AI Integration**: Multi-provider registry pattern (`server/services/ai-providers/`). Primary: Azure AI Foundry (`@azure-rest/ai-inference`) supporting GPT-5.4, GPT-5.2, GPT-4o. Fallback: Anthropic Claude Sonnet 4.5 via Replit AI Integrations. Active provider + model are DB-backed settings (`aiProvider`, `aiModel`) switchable from Admin Settings dialog. 90-day caching and content review workflow unchanged.
-   **Multi-Tenant Architecture**: ~60% complete. Tenant-private model visibility, OAuth client management, and SSO provisioning done. Branding and domain mapping remaining. See `PRODUCT_BACKLOG.md` for full status.
-   **In-App Support System**: Full-stack support ticket system with CRUD, replies, admin management console (Support tab in Admin sidebar), AI help chatbot (SSE streaming from user guide), What's New modal (AI-summarized changelog on login, system-level `showWhatsNew` toggle default OFF), Microsoft Planner integration (auto-sync tickets to tenant-configured plans, multi-tenant credentials). Public /help and /changelog pages render USER_GUIDE.md and CHANGELOG.md. Header help menu dropdown. Email notifications via SendGrid on ticket create/close. Schema tables: `supportTickets`, `supportTicketReplies`, `supportTicketPlannerSync`. Routes: `server/routes-support.ts`. Frontend: `UserGuide.tsx`, `Changelog.tsx`, `Support.tsx`, `WhatsNewModal.tsx`, `HelpChatPanel.tsx`, `SupportManagement.tsx`.
    - **What's New modal — dismiss bug fixes and crash fix (May 2026)**: Three bugs caused repeated modal appearances, plus one crash. (1) `WhatsNewModal` now uses local `dismissed` state + a ref guard so the dialog closes *immediately* on any dismiss action, preventing the async-mutation lag that let users think the close hadn't registered and caused `onOpenChange` to fire multiple times. (2) `detectChangelogVersion` now scans ALL `Version X.X` markers in CHANGELOG.md and returns the highest one, so new CHANGELOG sections without a version header no longer freeze the detected version. (3) The `GET /api/changelog/whats-new` route now reads the user record fresh from DB (instead of only from `req.user`) to guarantee the latest `lastDismissedChangelogVersion` is used even if session state is stale. (4) **Crash fix**: `WhatsNewModal` is rendered outside the `ErrorBoundary` in `App.tsx` (after the closing `</ErrorBoundary>` tag). When the AI summary generation failed and the fallback path returned a null/undefined `highlights` value, the old modal called `data.highlights.map(...)` unconditionally — a `TypeError` that crashed the entire app for any page the user was on, including `/courses/:slug`. Fixed by adding `data.highlights && data.highlights.length > 0 &&` guard before the map. Current changelog version: **2.1** (May 2026 section).
-   **Galaxy Client Portal API**: Two API tiers at `/api/galaxy/v1/*`. (1) **User-scoped OAuth API** — bearer-token auth via `oauthTokens`, requires `galaxy_portal` scope + tenant exposure policy enabled. Endpoints: `/me`, `/artifacts`, `/assessments`, `/assessments/:id`, `/insights/me`, `/courses`, `/certificates`, `/attestations`, plus write endpoints for course progress and attestation signing. Webhooks are HMAC-SHA256-signed. OpenAPI 3.1 spec at `/api/galaxy/v1/openapi.json`. (2) **Portal Key API** (`/api/galaxy/v1/portal/*`) — static API keys (`galaxyPortalKeys` table, SHA-256-hashed) for external portals like synozur-baseline. Auth via `Authorization: ApiKey <key>` or `X-Galaxy-Key` header. Domain-scoped: requests carry `?domain=synozur.com`; domain is resolved to a tenant via the `tenantDomains` table. Endpoints: `GET /portal/models?domain=` (published models for tenant), `GET /portal/courses?domain=` (public + tenant-shared published courses), `GET /portal/results?domain=` (anonymized aggregate results), `GET /portal/traffic?from=&to=` (app-wide traffic stats for central reporting). Keys are issued/revoked/managed in Admin → Galaxy Portal → Portal Keys tab (global admin only). `lastUsedAt` is updated on each call. CORS is enforced per-key via `allowedOrigins` array. Routes: `server/routes/galaxy/portal.ts`.
-   **Planner Integration (System-Level)**: All support tickets sync to a single Synozur Planner board, configured via system settings (`plannerEnabled`, `plannerPlanId`, etc.). Uses `AZURE_CLIENT_ID`/`AZURE_CLIENT_SECRET`/`AZURE_TENANT_ID` environment variables directly — no manual tenant ID entry needed. Global-admin-only configuration in Admin → Support → Settings. No per-tenant Planner setup needed.
-   **Learning Courses Module**: Course authoring + learner experience. Schema in `shared/schema.ts` (`courses`, `courseModules`, `lessons`, `courseEnrollments`, `lessonProgress`, `courseTags`, `courseTagAssignments`, `assessmentCourseLinks`, `attestationRecords`, `scormPackages`, `courseTenants`). Service layer in `server/services/course-service.ts` (NOT in `IStorage` to avoid bloat). Routes in `server/routes/course-routes.ts` registered via `registerCourseRoutes`. Lesson types: `rich_text`, `slides`, `video`, `audio`, `quiz`, `attestation`, `scorm`. Quiz scoring is server-side (`scoreQuiz`); attestations record signedName + IP/UA. Enrollments auto-recalc `progressPercent`/`status` from required-lesson completions. Admin UI at Admin → Content → Courses (`client/src/components/admin/CourseManagement.tsx`); learner pages at `/courses` and `/courses/:slug` (`client/src/pages/Courses.tsx`, `CourseDetail.tsx`). SCORM 1.2/2004 import + export, certificate PDF generation, attestation reminders, and AI-driven course recommendations from assessment results are tracked as follow-ups (the `assessment_course_links` table is wired but the recommendation surface is not yet rendered). **PR4 additions**: Courses now support archiving (status=`archived`, toggled from admin card — hidden from learners but preserved in DB), hero image upload via GCS, and tenant sharing (`courseTenants` junction, `visibility=private` + share list). Admin course cards show an archived toggle and a tenant-share dialog.
-   **Academies Module** (PR4): Sequenced learning paths that combine Orion-authored courses with external resources (LinkedIn Learning, Coursera, Pluralsight, YouTube, Udemy, edX, other). Schema: `academies` (slug, title, description, summary, imageUrl, estimatedMinutes, status, visibility, ownerTenantId), `academyTenants` (sharing junction), `academyItems` (ordered list of `course` or `external` items with provider, URL, duration). Service: `server/services/academy-service.ts`. Routes: `server/routes/academy-routes.ts`. Admin UI: `client/src/components/admin/AcademyManagement.tsx` (Admin → Content → Academies). Learner pages: `/academies` (browse) and `/academies/:slug` (detail). Visibility and tenant-sharing follow same pattern as courses. DB tables: `academies`, `academy_tenants`, `academy_items` — pushed to DB via `drizzle-kit push`.
-   **Course JSON Export/Import**: Portable `.orion-course.json` format. Export: `GET /api/courses/:id/export/json` → full course doc (metadata + modules + lessons + tag names; no enrollment data or SCORM binaries). Import: `POST /api/courses/import/json` → creates fresh course as `draft`, dedupes slug, resolves/creates tags by name. Service logic in `server/services/course-import-export.ts`. UI: Export (download icon) on each course card; Import button at top of Courses admin page triggers file picker. Sample course (`samples/sample-course.orion-course.json`) — "Maturity Model Assessment Fundamentals", 3 modules, 11 lessons, 3 tags — checked into repo and seeded in dev DB via `npx tsx scripts/seed-sample-course.ts`.

### System Design Choices
-   **Frontend Bundle Optimization**: Utilizes `React.lazy()` and dynamic `import()` for code-splitting to reduce initial bundle size. Pages are eagerly imported only for `Landing` and `not-found`, with all other pages lazy-loaded under a single `<Suspense>` boundary. Heavy libraries like `jspdf` and `recharts` are dynamically loaded only when needed by specific routes.
-   **Provisioning Settings**: Allows tenant self-creation and user self-provisioning via SSO with email domain matching. Includes public domain protection to block consumer domains from tenant creation.
